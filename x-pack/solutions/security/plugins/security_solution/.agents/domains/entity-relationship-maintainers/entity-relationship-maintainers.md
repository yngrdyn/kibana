# Domain: Entity Relationship Maintainers

## What every reviewer must know

This domain is the Entity Store v2 relationship-maintainer framework: a shared 2-step ES|QL/composite-aggregation engine (`engine/`) plus per-relationship, config-driven maintainers (`accesses`, `administers`, `communicates_with`, `owns`, `supervises`) that resolve raw signals from integration logs or entity `raw_identifiers` into canonical `entity.relationships.<relType>.ids` on actor entity documents, validate resolved targets against the entities-v2 index when `validateTargetIds` is set, and dual-write relationship observation metadata via `EntityMetadataClient.bulkAppendMetadata`. Each maintainer runs as a scheduled background task registered via the `entityStore` plugin's `RegisterEntityMaintainerConfig` contract; its `run()` callback receives `{ esClient, cpsEsClient?, logger, status, crudClient, entityMetadataClient, abortController }` and delegates to `runRelationshipMaintainer` with per-integration configs from `configs.ts`. Step 1 pages actors via composite aggregation against `config.indexPattern(namespace)`; Step 2 runs ES|QL per actor page; results are written via `crudClient.bulkUpdateEntity` and appended as metadata docs. Raw-identifiers maintainers (`administers`, `supervises`, `owns`) read from the latest entity index, use a `lastProcessedTimestamp` watermark for incremental processing, and require an explicit Task Manager `timeout` (e.g. `1h`) because the framework default of 5m is insufficient for full scans over 160M+ docs.

## Architectural invariants

- **Abort-signal errors must be detected and treated as a clean (non-error) exit — checked _before_ `isIndexNotFound`** — When the abort signal fires during `esClient.search`, the resulting error does not match `isIndexNotFound`, falls through to the generic error logger and re-throw, and crashes the maintainer, blocking future runs. (PR #266159, @macroscopeapp, `engine/run_relationship_maintainer.ts`)
- **An ES|QL query failure must surface as an error in the return metrics — never write partial data while reporting success** — Continuing with `break` after an ES|QL failure lets already-collected records be written while returning success metrics, so callers/operators cannot distinguish a partial run from a complete one. (PR #266159, @macroscopeapp)
- **EUID values in ES|QL must be computed via `euid.esql.getEuidEvaluation(...)`, not manual string concatenation** — Hand-built IDs like `CONCAT("user:", user.name)` bypass the canonical EUID path and force extracting far more fields than necessary; the evaluation helper is the canonical, minimal-extraction path. (PR #255418 @JordanSh; PR #262345 @niros1)
- **Empty/garbage target entity IDs must be filtered consistently for every entity-type variant in the ES|QL WHERE clause** — Guarding only one type (e.g. `Device`) while leaving another (e.g. `User`) unguarded produces invalid IDs such as `"user:@entra_id"` from an empty UPN, which is neither filtered nor a valid identity. (PR #262345, @macroscopeapp, `communicates_with/integrations/azure_auditlogs/build_esql_query.ts`)
- **The Task Manager `abortController` signal must be propagated through the entire maintainer run** — TM provides an abort controller when creating the task; ignoring it means the task cannot be cancelled cleanly. (PR #255418, @pmuellr, `accesses/index.ts`)
- **The maintainer registry must throw on an unknown ID, never return `undefined`** — A silent `undefined` from `get`/`getLifecycle` hides bugs and makes "not registered" indistinguishable from "not yet initialised". (PR #262317, @chennn1990, `entity_maintainers_registry.ts`)
- **The namespace must be validated against `NAMESPACE_PATTERN` at the engine boundary before any index-pattern interpolation** — `assertValidNamespace` runs at the top of `runRelationshipMaintainer` (`engine/run_relationship_maintainer.ts:405-420`; pattern `/^[a-z0-9][a-z0-9_-]{0,99}$/` in `engine/validate_namespace.ts`). This is a correctness guard (rejecting malformed namespaces), not a security/injection boundary.
- **The watermark must only advance after a non-aborted run** — `administers`/`supervises`/`owns` return the new `lastProcessedTimestamp` on success but return the prior `status.state` unchanged on abort (`administers/index.ts:48`). Only an abort blocks the watermark; a per-integration ES|QL failure does **not** prevent the watermark from advancing. Advancing on an aborted run would skip un-processed entities permanently.
- **Full-scan and raw-identifiers maintainers must set an explicit Task Manager `timeout` matched to expected run duration** — Leaving `timeout` unset inherits the framework default of 5m; for a daily full scan over a 160M+ doc store, 5m kills the first run before completion, and because the watermark only advances on success, the maintainer keeps re-scanning the same window. (@darnautov, PR #274261, `maintainers/supervises/index.ts`, `maintainers/owns/index.ts`, `maintainers/administers/index.ts`)
- **Task Manager saved-object state must persist only what the next run reads — `lastProcessedTimestamp`** — Spreading the full engine `RunResult` (totalBuckets, totalWritten, truncated, redundant `lastRunTimestamp`, …) into TM state bloats saved objects; only `lastProcessedTimestamp` is consumed on the next incremental run. (@darnautov, PR #274261, `maintainers/owns/index.ts`, `maintainers/supervises/index.ts`)
- **Dual-write metadata append must filter to the same validated and successfully-written target set as `writeEntityIds`** — When `validateTargetIds` is set, `writeEntityIds` prunes dangling/unresolved targets from the latest index; appending metadata for the unfiltered target set records relationships the latest index rejected, causing latest-index vs history datastream divergence. (@seanrathier, PR #271806, `engine/run_relationship_maintainer.ts:265-304`)
- **Raw-identifiers maintainers must union all candidate identifier fields into EUID candidates, then validate every candidate against `.entities.v2.*` before writing** — The canonical EUID source field is not known a priori (email vs id vs name); picking a single field produces dangling IDs. Only targets passing `LOOKUP`/validation with `validateTargetIds: true` may be written. (@alexreal1314, PR #274261, `maintainers/supervises/configs.ts`)
- **Relationship metadata writes must not go through the generic `CRUDClient`** — Metadata append and relationship-list logic belongs in dedicated clients (`EntityMetadataClient`); inserting it into the shared CRUD API leaks maintainer-specific behavior to all CRUD consumers. (@romulets, PR #271806, `entity_store/server/domain/crud/crud_client.ts`)
- **Raw Elasticsearch I/O must stay in `infra/elasticsearch/` — domain clients delegate, never call `esClient` directly** — Direct ES calls inside domain clients (e.g. CRUD client) bypass the infra isolation layer and make I/O paths harder to test and reason about. (@romulets, PR #271806, `entity_store/server/domain/crud/crud_client.ts`)
- **Metadata append path must default to `refresh: false`** — Relationship metadata is written to an append-only background datastream; waiting for refresh on every bulk append adds unnecessary latency with no caller requiring synchronous read-after-write. (@romulets, PR #271806, `entity_store/server/infra/elasticsearch/entity_metadata.ts`)
- **Scout maintainer API suites must call `waitForEntityStoreRunning` after install and set `apiTest.setTimeout(180_000)`** — Starting test bodies immediately after install races entity-store initialization and reproduces the flakiness the host-target sibling suite already works around. (@MadameSheema, PR #274261, `test/scout/.../maintainers/user_target_raw_identifiers_maintainer_suite.spec.ts`)

## Common review patterns (learned from real PRs)

- **Omit unchanged relationship fields with `undefined`, not `{ ids: [] }`** — Setting an empty array explicitly clears the field and erases relationships observed in an earlier window but absent in the current one; `undefined` omits the field so prior data is preserved. (@tiansivive, PR #262911, `accesses/update_entities.ts`)
- **Prefer `satisfies SomeType` over `as SomeType`** — `satisfies` validates the object shape without widening it; flagged on update-entity functions. (@tiansivive, PR #262911)
- **Create `esClient` and `cpsEsClient` unconditionally; let each maintainer choose** — Do not gate CPS client creation behind an `isServerless` flag at the execution layer; the decision belongs to each maintainer's data-access intent. (@uri-weisman, PR #269578, `entity_store/server/tasks/entity_maintainers/execution.ts`)
- **One registry map with typed `task` / `lifecycle` sub-properties — not two separate maps** — (@chennn1990, PR #262317, `entity_maintainers_registry.ts`)
- **Fail fast on a missing namespace; do not fall back to `""`** — Keep the `currentStatus?.namespace || currentStatus?.metadata?.namespace` lookup chain and let it surface early. (@chennn1990, PR #262317, `execution.ts`)
- **`autoStart` defaults to `true`; `autoStart: false` is the exceptional, test-only path** — Renamed from a no-default `enabled` flag. (@chennn1990, PR #262317, `utils/validator.ts`)
- **`extends` an existing interface instead of re-declaring its fields** — (@chennn1990, PR #262317, `entity_maintainers_client.ts`)
- **Deduplicate lookback-filter construction into the shared `buildLookbackFilter`** — Used in both `run_relationship_maintainer.ts` and `build_actor_discovery_query.ts`. (@maxcold, PR #272948)
- **Log a run/skip decision once, at the point of decision — not again in the outer loop** — (@alexreal1314, PR #269578, `engine/run_relationship_maintainer.ts`)
- **Comments explain _why_ for non-obvious decisions; "future work" comments must link a ticket** — (@maxcold, PR #272948, `administers/configs.ts`)
- **Keep synchronous-run API execution separate from Task Manager task code** — Extract `executeMaintainerRun` / `runSync` into a dedicated `execution.ts`; the API route calls it directly, bypassing TM status/licensing checks that don't apply to sync API calls. (@chennn1990, PR #262317, `entity_store/server/tasks/entity_maintainers/index.ts`)
- **Use `esClient.helpers.bulk` for high-fan-out metadata bulk writes** — Raw `esClient.bulk` materializes the entire operations array; the helper chunks by `flushBytes`, streams via a generator, retries 429s, and supports an `onDrop` callback for drop-reason logging. (@opauloh, PR #271806, `entity_store/server/infra/elasticsearch/entity_metadata.ts`)
- **Do not expose index-name helpers for the metadata datastream** — Metadata lives in a datastream/write alias, not a concrete index; dead `getEntityMetadataIndexName` helpers and "index" naming mislead callers. (@romulets, PR #271806, `entity_store/common/domain/entity_index.ts`)
- **Do not leak internal implementation details into customer-facing copy** — Fold internal artifacts (e.g. "entity relationship history") under existing customer concepts ("Security Entity store records"). (@romulets, PR #271806, `public/entity_analytics/.../clear_entity_data_button.tsx`)
- **IDP user EUIDs are namespace-suffixed; host targets are namespace-less** — User actors use `CONCAT("user:", <value>, "@<namespace>")` (e.g. `user:<id>@okta`); host targets use `host:<name>`. Null-safe `MV_APPEND` guards are required because `MV_APPEND(null, x)` returns null. (PR #274261, `maintainers/supervises/configs.ts`, `maintainers/administers/configs.ts`)
- **New host-target raw-identifiers maintainers need their own Scout spec file** — `scout_max_one_describe` means a second `registerHostTargetRawIdentifiersMaintainerSuite` call in the same file fails lint; do not rely on a "reuse" comment in an existing spec. (@MadameSheema, PR #274261)

## Security considerations

- **Namespace validation (`assertValidNamespace`) is a correctness guard, not a security boundary** — it rejects malformed namespaces before index-pattern interpolation (`engine/validate_namespace.ts`, enforced at `engine/run_relationship_maintainer.ts:405-420`), but is not relied upon as an index-pattern/injection defense.
- **Account-exclusion gaps are a known data-quality (not yet security-enforced) backlog** — Windows machine accounts (`WORKSTATION$`), Linux service accounts (root, bin, daemon, sshd, etc.), and network logon type 3 (machine-to-machine) are not currently excluded from accesses queries; authors deferred these. A reviewer adding a new integration should consider whether non-human/service identities pollute relationship data. (PR #255418, @niros1)
  > **[VERIFY]:** Should machine/service account exclusion be a hard requirement for new accesses/communicates_with integrations, or remain per-integration discretion? (unconfirmed)
- **Scout maintainer install routes require `securitySolution` + entity-analytics privileges that lower test roles lack** — Tests calling the entity-maintainers init/run routes must use an admin-capable role; omitting this in test comments hides why elevated privileges are required. (@MadameSheema, PR #274261)

## Performance constraints

- **Step 1 pages actors via composite aggregation at `COMPOSITE_PAGE_SIZE = 3500` per page** (`engine/constants.ts`, `engine/build_actor_discovery_query.ts`); each page becomes an OR-of-`terms` DSL filter narrowing the Step 2 ES|QL query (`build_actor_discovery_query.ts:142`). Adding actor fields or relaxing actor-presence filters widens the candidate set and increases ES|QL load per page. `COMPOSITE_PAGE_SIZE` is tuned against source **log** volume (typically higher than entity count), not a fixed entity-count ceiling — entity volumes stay well below log volumes; large customers reach on the order of ~1M entities, usually fewer.
- **An ES|QL failure for one integration currently stops the entire maintainer run** — no subsequent integration writes its results; per-integration error isolation was suggested but is unresolved. (PR #255418, @niros1)
- **Full-scan maintainers over 160M+ docs require explicit `timeout: '1h'` (or similar) — not the 5m TM default** — A timed-out first run leaves the watermark unchanged, causing repeated full re-scans of the same window on every daily schedule. (@darnautov, PR #274261, `maintainers/supervises/index.ts`, `maintainers/owns/index.ts`, `maintainers/administers/index.ts`)
- **Incremental raw-identifiers maintainers advance `lastProcessedTimestamp` only on successful completion** — Watermark-based incremental processing via `entity.lifecycle.last_seen > lastProcessedTimestamp` (`engine/build_raw_identifiers_query.ts:87`, `administers/index.ts:32-35`) avoids re-processing the entire entity index each run; a failed or timed-out run must not advance the watermark.
- **Metadata bulk append uses `refresh: false` by default** — Background-written append-only datastream records do not require synchronous refresh; adding refresh waits would multiply write latency across high-fan-out actor→target pairs. (@romulets, PR #271806)
- **High-fan-out metadata writes should use `esClient.helpers.bulk` with `flushBytes` chunking** — Prevents exceeding `http.max_content_length` on large maintainer runs. (@opauloh, PR #271806)
- **`disableLookbackWindow` is fully wired into both the composite-agg and ES|QL query builders** (as of follow-up PR #272948) — raw-identifiers maintainers gate out the lookback-window time filter in the actual queries, not only in the `Maintainer.lookback_window` metadata field.

## Historical catches

- [PR #266159](https://github.com/elastic/kibana/pull/266159) — Reviewer caught that an abort-signal error during `esClient.search` fell through to the generic error handler and crashed the maintainer — A generic reviewer would not know abort must be matched _before_ `isIndexNotFound` and treated as a clean exit, not a failure.
- [PR #266159](https://github.com/elastic/kibana/pull/266159) — Reviewer caught that `break`-on-ES|QL-failure wrote already-collected records while returning success metrics — A generic reviewer would not realize the function reports success with no signal that the write was partial.
- [PR #262345](https://github.com/elastic/kibana/pull/262345) — Reviewer caught that an empty UPN produced an invalid `"user:@entra_id"` target ID because the empty-field guard covered only `Device`, not `User` — A generic reviewer would not know each entity-type branch needs its own empty-field guard in the ES|QL WHERE clause.
- [PR #262317](https://github.com/elastic/kibana/pull/262317) — Reviewer held firm that the maintainer registry must throw on an unknown ID rather than return `undefined` — A generic reviewer would not see why distinguishing "not registered" from "not yet initialised" matters for this lifecycle.
- [PR #266159](https://github.com/elastic/kibana/pull/266159) — `customActor` configs were silently filtered to zero buckets because the Step 1 base filter still required standard `user.*` ECS fields (notably `azure_auditlogs`) — A generic reviewer would not know that supplying `config.customActor` must bypass the ECS actor-presence filter, or the maintainer silently writes nothing.
